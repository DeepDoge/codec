# Migration Guide

Breaking changes introduced in `@nomadshiba/codec` **0.3.0**.

---

## `stride` is now a discriminated union

The `stride` property on all codecs changed from a plain `number` to a `Stride` discriminated-union type exported from the package.

| Kind     | Shape                             | Meaning                                |
| -------- | --------------------------------- | -------------------------------------- |
| Fixed    | `{ kind: "fixed"; size: number }` | Always encodes to exactly `size` bytes |
| Variable | `{ kind: "variable" }`            | Encoded size depends on the value      |

**Before:**

```ts
if (codec.stride >= 0) { /* fixed */ }
if (codec.stride < 0) { /* variable */ }
const size = codec.stride; // number
```

**After:**

```ts
import type { Stride } from "@nomadshiba/codec";

if (codec.stride.kind === "fixed") { const { size } = codec.stride; }
if (codec.stride.kind === "variable") { /* variable */ }
```

Custom codec implementations must update their `stride` declarations:

```ts
// Before
class MyCodec extends Codec<MyType> {
	readonly stride = 4;
}

// After
import { Codec, Stride } from "@nomadshiba/codec";
class MyCodec extends Codec<MyType> {
	readonly stride: Stride<"fixed"> = { kind: "fixed", size: 4 };
}
```

## `UnionCodec` renamed back to `EnumCodec`

`UnionCodec` (introduced in 0.2.0) has been renamed to `EnumCodec`. All associated types were renamed accordingly.

| Before           | After           |
| ---------------- | --------------- |
| `UnionCodec<T>`  | `EnumCodec<T>`  |
| `UnionGeneric`   | `EnumGeneric`   |
| `UnionInput<T>`  | `EnumInput<T>`  |
| `UnionOutput<T>` | `EnumOutput<T>` |
| `UnionOptions`   | `EnumOptions`   |

```ts
// Before
import { StringCodec, U8, UnionCodec } from "@nomadshiba/codec";
const Event = new UnionCodec({ Click: U8, Message: new StringCodec() });

// After
import { EnumCodec, StringCodec, U8 } from "@nomadshiba/codec";
const Event = new EnumCodec({ Click: U8, Message: new StringCodec() });
```

---

## `EnumCodec` variant indices are now in definition order

Previously `UnionCodec` (now `EnumCodec`) sorted variant names alphabetically before assigning indices. Indices are now assigned in **definition order**.

This is a **wire-format breaking change** if your variant names are not already in alphabetical order.

```ts
const Event = new EnumCodec({ Click: U8, Message: new StringCodec() });
// Before: "Click" → 0, "Message" → 1  (happened to match alphabetical)
// After:  "Click" → 0, "Message" → 1  (same here, but order now matters)

const Event2 = new EnumCodec({ Message: new StringCodec(), Click: U8 });
// Before: "Click" → 0, "Message" → 1  (alphabetical, ignoring definition order)
// After:  "Message" → 0, "Click" → 1  (definition order)
```

Re-order your variant definitions to match the alphabetical order you relied on, or re-encode your existing data.

---

## `EnumOptions.indexCodec` renamed to `indexer`

```ts
// Before
new UnionCodec({ ... }, { indexCodec: U16 });

// After
new EnumCodec({ ... }, { indexer: U16 });
```

---

## `ArrayCodec`: `codec` → `item`, `countCodec` → `counter`

| Before                               | After                             |
| ------------------------------------ | --------------------------------- |
| `arrayCodec.codec`                   | `arrayCodec.item`                 |
| `new ArrayCodec(el, { countCodec })` | `new ArrayCodec(el, { counter })` |

```ts
// Before
const arr = new ArrayCodec(U16, { countCodec: U32 });
arr.codec; // element codec

// After
const arr = new ArrayCodec(U16, { counter: U32 });
arr.item; // element codec
```

---

## `TupleCodec`: `codecs` → `items`

```ts
// Before
tupleCodec.codecs;

// After
tupleCodec.items;
```

---

## `StructCodec`: `countCodec` option removed (was unused)

No action needed unless you were inspecting internal properties.

---

## `MappingCodec`: `countCodec` → `counter`; `entryCodec` returns updated shape

```ts
// Before
new MappingCodec([Str, U8], { countCodec: U32 });

// After
new MappingCodec([Str, U8], { counter: U32 });
```

The `entryCodec` getter now returns `TupleCodec.items` (previously `TupleCodec.codecs`) internally — no change in the returned value type, but the path through
the internal `TupleCodec` changed.

## `StringCodec`: `lengthCodec` → `sizer`

| Before                                  | After                             |
| --------------------------------------- | --------------------------------- |
| `new StringCodec({ lengthCodec: U32 })` | `new StringCodec({ sizer: U32 })` |
| `stringCodec.lengthCodec`               | `stringCodec.sizer`               |

## `BytesCodec`: `lengthCodec` → `sizer`

| Before                                 | After                            |
| -------------------------------------- | -------------------------------- |
| `new BytesCodec({ lengthCodec: U32 })` | `new BytesCodec({ sizer: U32 })` |
| `bytesCodec.lengthCodec`               | `bytesCodec.sizer`               |

## Removed deprecated type aliases

The following deprecated aliases (introduced as shims in earlier versions) have been removed entirely:

| Removed           | Use instead        |
| ----------------- | ------------------ |
| `TupleValue<T>`   | `TupleOutput<T>`   |
| `StructValue<T>`  | `StructOutput<T>`  |
| `ArrayValue<T>`   | `ArrayOutput<T>`   |
| `UnionValue<T>`   | `EnumOutput<T>`    |
| `MappingValue<T>` | `MappingOutput<T>` |

## New: `PaddedEnumCodec`

`PaddedEnumCodec` is a fixed-size enum codec. All variants must be fixed-size; shorter variant payloads are zero-padded to the max variant size, so every
encoded value has the same byte length. Advertises `Stride<"fixed">`, enabling tighter downstream optimisations and use inside other fixed-size codecs.

```ts
import { PaddedEnumCodec, U32, U8 } from "@nomadshiba/codec";

const Tag = new PaddedEnumCodec({ A: U8, B: U32 }); // stride = { kind: "fixed", size: 5 }
//   (1 index byte + 4 max payload, A's 1-byte payload is zero-padded to 4)
```

## Package entry point moved to `lib/mod.ts`

The package root was reorganised. All source now lives under `lib/`. The JSR `exports` field was updated accordingly.

If you import the package via its JSR name (`@nomadshiba/codec`) **nothing changes** — JSR resolves the entry point automatically. If you were importing
directly from the raw file path, update your imports:

```ts
// Before
import { U32 } from "@nomadshiba/codec";

// After
import { U32 } from "@nomadshiba/codec";
```

---

## New: `ModelCodec`

`ModelCodec` is a named-field struct codec where individual fields can be marked **optional** by appending `?` to the key in the shape definition.

Optional fields are prefixed by a single presence byte on the wire (`0x01` = present, `0x00` = absent). When absent the field value is omitted from the byte
stream entirely.

```ts
import { ModelCodec, U8, Utf8 } from "@nomadshiba/codec";

const User = new ModelCodec({
	name: Utf8,
	"nickname?": Utf8,
	age: U8,
});

// Encode without the optional field
const bytes = User.encode({ name: "Alice", age: 30 });

// Encode with the optional field present
const bytes2 = User.encode({ name: "Alice", nickname: "Ali", age: 30 });
```

Use `ModelCodec.partial()` to produce a fully-optional variant of an existing model shape.

---

## New utility types: `FixedCodec`, `VariableCodec`

Two narrowing types are now exported for use in generic constraints:

```ts
import type { FixedCodec, VariableCodec } from "@nomadshiba/codec";

function readFixed<C extends FixedCodec>(codec: C, count: number) { ... }
function writeLengthPrefixed<C extends VariableCodec>(codec: C, value: C["_INPUT_"]) { ... }
```

---

## New base class: `CodecWithOffsets`

`CodecWithOffsets<O, I, Offsets>` extends `Codec` and adds `encodeWithOffsets` / `decodeWithOffsets` methods that return a per-field byte-offset map alongside
the encoded/decoded value. Useful for struct-like codecs when callers need to know where each field sits inside the buffer.

---

## New namespace utilities: `Codec.InferInput`, `Codec.InferOutput`

Two phantom-type helpers are now available on the `Codec` namespace for extracting a codec's input/output types without triggering infinite-recursion issues
with `infer`:

```ts
import type { Codec, U32Codec } from "@nomadshiba/codec";

type In = Codec.InferInput<U32Codec>; // number
type Out = Codec.InferOutput<U32Codec>; // number
```
